# nc (netcat) 상세 설명

`nc`는 "네트워크의 스위스 아미 나이프"라고 불릴 만큼 다양한 용도로 쓰이는 TCP/UDP 네트워크 유틸리티입니다. 이 프로젝트의 `watcher.sh`에서는 서버 상태를 확인(헬스체크)하고 장애 알림을 보내는 용도로 사용됩니다. 이 문서는 `nc`의 개념, 옵션, 그리고 이 프로젝트에서의 실제 사용법을 상세히 설명합니다.

## 1. nc란 무엇인가

`nc`(netcat)는 TCP 또는 UDP 연결을 만들어 데이터를 주고받는 커맨드라인 도구입니다. 파일을 다루듯 네트워크 소켓을 다룰 수 있게 해주며, 표준입력(stdin)으로 받은 데이터를 네트워크로 보내고, 네트워크로부터 받은 데이터를 표준출력(stdout)으로 내보냅니다. 그래서 파이프(`|`)와 결합해서 쓰기에 매우 적합합니다.

대표적인 활용 사례는 다음과 같습니다.

- 특정 호스트의 포트가 열려 있는지 확인 (포트 스캔/헬스체크)
- 간단한 TCP 클라이언트/서버 구현 (채팅, 파일 전송)
- 셸 스크립트 안에서 HTTP 요청을 raw하게 흉내 내기 (이 프로젝트가 하는 방식)
- 디버깅 시 특정 포트로 임의의 데이터를 보내보기

주의: 이 이미지는 `busybox`를 베이스로 하므로, GNU netcat이나 nmap의 ncat이 아니라 **BusyBox에 내장된 경량 nc**를 사용합니다. 옵션 지원 범위가 더 좁습니다.

## 2. 기본 문법

```
nc [옵션] HOST PORT      # 클라이언트 모드: HOST의 PORT로 접속
nc [옵션] -l -p PORT      # 서버(리슨) 모드: PORT에서 연결을 기다림
```

## 3. BusyBox nc의 주요 옵션

이 이미지(`busybox:1.36`)의 `nc`가 지원하는 옵션은 다음과 같습니다 (`nc` 를 인자 없이 실행하면 확인 가능).

| 옵션 | 설명 |
|---|---|
| `-e PROG` | 연결이 맺어진 후 `PROG`를 실행합니다. 반드시 옵션 목록의 마지막에 위치해야 합니다. 셸을 붙여서 원격 접속처럼 쓰는 데 주로 사용됩니다 (예: 리버스 셸). |
| `-l` | 리슨(listen) 모드로 동작합니다. 즉 서버처럼 들어오는 연결을 기다립니다. |
| `-lk` | `-e`와 함께 사용하면, 연결이 끊겨도 서버가 종료되지 않고 계속 새 연결을 받는 지속 서버(persistent server)로 동작합니다. |
| `-p PORT` | 로컬(자신이 사용할) 포트를 지정합니다. |
| `-s ADDR` | 로컬 주소를 지정합니다. 여러 네트워크 인터페이스가 있을 때 특정 IP로 바인딩하고 싶을 때 사용합니다. |
| `-w SEC` | 연결 시도 및 마지막 읽기 작업에 대한 타임아웃(초)을 지정합니다. |
| `-i SEC` | 데이터를 한 줄씩 보낼 때 줄 사이의 지연 시간(초)을 지정합니다. |
| `-n` | DNS 조회를 하지 않습니다 (호스트명이 아니라 IP 주소만 다룰 때 속도를 위해 사용). |
| `-u` | UDP 모드로 동작합니다 (기본은 TCP). |
| `-b` | 브로드캐스트를 허용합니다. |
| `-v` | 상세(verbose) 출력을 켭니다. 연결 성공/실패 등을 표준에러로 알려줍니다. |
| `-o FILE` | 송수신하는 트래픽을 헥사 덤프(hex dump) 형태로 `FILE`에 기록합니다. |
| `-z` | Zero-I/O 모드. 데이터를 실제로 주고받지 않고 연결 가능 여부만 확인합니다 (포트 스캔에 유용). |

> 참고: BusyBox `nc`에는 GNU netcat에 흔히 있는 `-w`(대문자 `-W`), `-c`(명령 실행), TLS 지원 등이 없습니다. 위 표가 이 이미지에서 실제로 쓸 수 있는 옵션의 전부입니다.

## 4. 이 프로젝트에서의 사용법

### 4.1 헬스체크: `timeout 2 nc insideweb 80`

`watcher.sh`의 핵심 로직입니다.

```sh
printf "GET / HTTP/1.0\r\n\r\n" | timeout 2 nc insideweb 80 | grep -q '200 OK'
```

동작 순서는 다음과 같습니다.

1. `printf`가 raw한 HTTP GET 요청 문자열(`GET / HTTP/1.0\r\n\r\n`)을 생성해 표준출력으로 내보냅니다.
2. 파이프(`|`)를 통해 이 문자열이 `nc insideweb 80`의 표준입력으로 전달됩니다.
3. `nc insideweb 80`은 `insideweb`라는 호스트(참고: [README.md](./README.md)에 설명된 `--link` 별칭)의 80번 포트로 TCP 연결을 맺고, 표준입력으로 받은 HTTP 요청 문자열을 그대로 전송합니다.
4. 서버가 응답하면 그 응답(HTTP 상태줄 + 헤더 + 바디)이 `nc`의 표준출력으로 흘러나옵니다.
5. `timeout 2`가 이 전체 과정을 2초로 제한해서, 서버가 응답하지 않을 때 스크립트가 무한정 멈추는 것을 방지합니다.
6. 마지막으로 `grep -q '200 OK'`가 응답 안에 `200 OK` 문자열이 있는지 검사해서, 있으면 성공(종료 코드 0)으로 처리합니다.

이렇게 `nc`를 HTTP 클라이언트처럼 사용하는 방식은 정식 HTTP 클라이언트(`curl`, `wget`)가 없는 최소한의 BusyBox 환경에서 널리 쓰이는 기법입니다. 단, HTTP를 제대로 구현한 것은 아니라 리다이렉트, chunked 인코딩, HTTPS 등은 처리하지 못합니다. 단순히 "포트가 열려 있고 200을 반환하는가"만 확인하는 용도로는 충분합니다.

### 4.2 장애 알림 전송: `nc insidemailer 33333`

```sh
printf "To: admin@work Message: The service is down!" | nc insidemailer 33333
```

헬스체크가 실패했을 때, `insidemailer` 호스트의 33333번 포트로 평문 메시지를 전송합니다. 이 포트는 `mailer` 컨테이너가 리슨하고 있는 포트이며(`docker run --link mailer:insidemailer` 참고), `mailer` 쪽에서 이 데이터를 받아 실제 메일 발송 등의 처리를 한다고 가정합니다. 여기서도 `nc`는 단순히 "표준입력을 그대로 TCP로 전송하는 파이프" 역할을 합니다.

## 5. 자주 헷갈리는 점

- **`nc HOST PORT`에서 인자가 비어있으면 usage 메시지가 출력됩니다.** 이 프로젝트에서 과거에 `$INSIDEWEB_PORT_80_TCP_ADDR` 같은 환경변수를 참조하다가, 최신 Docker에서는 `--link`가 더 이상 이런 환경변수를 주입하지 않아 값이 비어 `nc`가 인자 없이 호출되며 usage 메시지만 반복 출력되는 문제가 있었습니다. 지금은 `--link`가 실제로 등록하는 `/etc/hosts` 별칭(`insideweb`, `insidemailer`)을 직접 사용하도록 고쳐서 해결했습니다.
- **`nc`는 연결이 끝나면(peer가 소켓을 닫으면) 종료됩니다.** HTTP/1.0 서버는 응답 후 연결을 닫아주므로 별도로 `-w`를 주지 않아도 `timeout`이 안전망 역할을 합니다.
- **BusyBox `nc`에는 `-q`(연결 종료 후 대기 시간) 옵션이 없는 빌드도 있습니다.** 대신 이 프로젝트는 `-w`/`timeout`으로 타임아웃을 제어합니다.
