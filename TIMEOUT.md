# timeout 커맨드 상세 설명

`timeout`은 뒤에 지정한 명령어를 정해진 시간 동안만 실행하고, 시간이 초과되면 강제로 종료시키는 커맨드입니다. `watcher.sh`에서는 `nc`가 응답 없는 서버 때문에 무한정 멈추는 것을 막기 위해 사용됩니다. 이 문서는 `timeout`의 개념, 문법, 옵션, 그리고 이 프로젝트에서의 실제 동작을 상세히 설명합니다.

## 1. timeout이란 무엇인가

셸 스크립트에서 외부 명령(네트워크 요청, DB 조회, 파일 다운로드 등)을 호출하면, 그 명령이 응답 없이 계속 대기(hang)할 위험이 있습니다. 특히 `nc`처럼 상대방이 연결을 닫아주기 전까지 계속 데이터를 기다리는 명령은, 서버가 죽어있거나 네트워크가 막혀 있으면 영원히 끝나지 않을 수 있습니다.

`timeout`은 이런 상황을 방지하기 위한 "안전장치(watchdog)"입니다. 지정한 시간이 지나면 실행 중인 명령에 시그널을 보내 강제로 종료시키고, 스크립트가 다음 단계로 넘어갈 수 있게 해줍니다.

## 2. 기본 문법

```
timeout [옵션] DURATION COMMAND [ARG]...
```

- `DURATION` : 허용할 최대 실행 시간. 초 단위 숫자를 기본으로 하며, 뒤에 단위 접미사를 붙일 수 있습니다.
  - `s` : 초 (기본값, 생략 가능) — 예: `2`, `2s`
  - `m` : 분 — 예: `1m`
  - `h` : 시간 — 예: `1h`
  - `d` : 일 — 예: `1d`
- `COMMAND [ARG]...` : 실제로 실행할 명령과 그 인자들.

## 3. 동작 원리

1. `timeout`은 `COMMAND`를 자식 프로세스로 실행합니다.
2. `DURATION`만큼 시간이 지나기 전에 `COMMAND`가 스스로 끝나면, `timeout`은 그 명령의 종료 코드를 그대로 반환합니다.
3. `DURATION`이 지나도 `COMMAND`가 끝나지 않았다면, `timeout`은 그 프로세스에 시그널(기본값 `SIGTERM`, 15번)을 보내 종료를 요청합니다.
4. 프로세스가 `SIGTERM`을 받고도 종료되지 않는 경우를 대비해, `-k`(kill-after) 옵션으로 일정 시간 뒤 `SIGKILL`(9번, 무조건 종료)을 추가로 보낼 수 있습니다 (BusyBox 빌드에 따라 지원 여부가 다를 수 있습니다).

## 4. 종료 코드(Exit Code)의 의미

`timeout`의 종료 코드는 단순히 "성공/실패"가 아니라 상황에 따라 다른 의미를 가집니다.

| 종료 코드 | 의미 |
|---|---|
| `COMMAND`가 정상 실행되어 반환한 코드 | 시간 내에 명령이 끝난 경우, 그 명령 자체의 종료 코드를 그대로 전달 |
| `124` | 시간 초과로 `timeout`이 명령을 강제 종료시킨 경우 |
| `125` | `timeout` 자신의 실행이 실패한 경우 (예: 옵션 오류) |
| `126` | `COMMAND`를 찾았지만 실행할 수 없는 경우 (권한 문제 등) |
| `127` | `COMMAND`를 찾을 수 없는 경우 |

> `watcher.sh`에서는 `timeout 2 nc ... | grep -q '200 OK'`처럼 파이프로 연결되어 있어서, `if` 조건문이 최종적으로 보는 종료 코드는 파이프의 **마지막 명령인 `grep -q`의 결과**입니다. 즉 `timeout`이 124(시간초과)를 반환하더라도, `nc`가 아무 응답도 못 내보냈다면 `grep -q '200 OK'`는 매치 실패(1)를 반환하므로 어차피 `else`(장애 알림) 분기로 자연스럽게 이어집니다.

## 5. BusyBox timeout의 주요 옵션

이 이미지(`busybox:1.36`)의 `timeout`은 GNU coreutils의 `timeout`보다 가벼운 버전입니다. 주요 옵션은 다음과 같습니다.

| 옵션 | 설명 |
|---|---|
| `-s SIG` | 시간 초과 시 보낼 시그널을 지정합니다. 기본값은 `TERM`입니다. 예: `-s KILL`, `-s 9` |
| `-t SEC` | (일부 BusyBox 빌드) 시간을 별도 옵션으로 지정. 최신 빌드에서는 보통 위치 인자(`DURATION`)로 대체됨 |
| `-k DURATION` | `SIGTERM`을 보낸 뒤에도 프로세스가 살아있으면, `DURATION` 만큼 더 기다렸다가 `SIGKILL`을 추가로 보냅니다 |

`nc` 자신도 응답성이 낮은 프로그램을 강제로 죽이기 어려운 경우가 있으므로(시그널을 무시하도록 짜여 있다면), 아주 견고하게 만들려면 `-k`를 함께 쓰는 것이 좋습니다. 다만 이 프로젝트의 `nc`는 단순 클라이언트라 기본 `SIGTERM`으로 충분히 종료됩니다.

## 6. 이 프로젝트에서의 사용법

```sh
printf "GET / HTTP/1.0\r\n\r\n" | timeout 2 nc insideweb 80 | grep -q '200 OK'
```

- `timeout 2 nc insideweb 80` : `nc insideweb 80`을 최대 **2초** 동안만 실행합니다.
- `nc`는 `insideweb`(웹 서버 컨테이너의 `--link` 별칭) 80번 포트에 연결해서, 표준입력으로 받은 HTTP 요청을 보내고 응답을 기다립니다.
- 정상적인 웹 서버라면 응답 후 바로 연결을 닫아주므로 2초 안에 끝나는 것이 보통입니다. 하지만 서버가 다운되어 있거나(TCP handshake 자체가 안 되거나), 응답은 하되 연결을 닫지 않는 이상한 상태라면 `nc`가 계속 대기하게 됩니다.
- 이때 `timeout`이 2초 뒤 `nc`를 강제 종료시켜, `while` 루프가 다음 반복으로 넘어갈 수 있도록 보장합니다.

이전 버전의 스크립트는 `nc -w 2 ...`처럼 `nc` 자체의 타임아웃 옵션(`-w`)을 사용했었는데, `-w`는 "연결 시도"와 "마지막 읽기"에 대한 타임아웃이라 상황에 따라 정확히 원하는 시점에 끊어지지 않을 수 있습니다. 반면 `timeout`은 프로세스 전체 실행 시간을 외부에서 강제로 제한하기 때문에 더 예측 가능하고 확실합니다. 그래서 현재 스크립트는 `nc -w` 대신 `timeout`을 앞에 붙이는 방식으로 바뀌었습니다.

## 7. 왜 mailer 쪽 `nc`에는 timeout이 없는가

```sh
printf "To: admin@work Message: The service is down!" | nc insidemailer 33333
```

장애 알림을 보내는 이 줄에는 `timeout`이 붙어있지 않습니다. 이는 의도적인 설계라기보다, `mailer` 컨테이너가 항상 응답성 있게 동작한다고 가정하고 있는 것입니다. 만약 `mailer`도 응답이 없거나 연결이 오래 걸릴 가능성이 있다면, 동일하게 `timeout N nc insidemailer 33333`으로 감싸주는 것이 더 견고합니다.
